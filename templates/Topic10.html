Automated Testing


Automated testing is using a program (or program suite) that provides correct and complete input to the program being tested, runs the program, and compares the program output to what was expected. Once the automated test is written, it can be run automatically whenever desired. Automated testing plays a major role in online systems, and the increasing corporate use of the Internet means an increase in the use of automated testing.
Obviously any kind of testing can be automated, but one of the most popular automations is load testing. Load testing is testing how a system will respond to a heavy load of activity. Load testing simulates the effect of having many users on an online system at one time and is very common with eApplications. It's also called stress testing and performance testing and is carried on throughout the life of the online system. This is why automation is so popular. A load test when a Web application is initially installed might return results that say the system works beautifully. This might not be true six months later as activity on the site increases. Load tests are run regularly so the IT staff can note the increasing traffic and respond to it before the site crashes - not after.
Automated test suites can be purchased or written by internal staff. Some automated tests must be individually written, as the system specifics can only be thoroughly tested with exact test data and test scenarios. In-house developers or consultants will be needed to write the automated tests. Vendor-supplied test suites are also growing in use, and products such as Compuware's QACenter, Rational's TeamTest, and Segue's SilkTest are popular.
Everyone has experienced the frustration of working with a program with errors (bugs), and we all wonder how the vendor let the product be marketed before it worked. It's nice to know that vendors and corporations alike are spending more time and money on testing.
QA (Quality Assurance), SQA (Software Quality Assurance)
Quality assurance is part of any product creation - including the creation of software systems. It is the development of standards, and formal monitoring to make sure that the standards are followed. These standards are used to develop products, including software systems, and are monitored to ensure that the system does what it is supposed to do. While the standards must be set in the beginning of the development life cycle, quality assurance is often considered to be part of testing as this is the most common QA activity. Complete quality assurance, however, also includes other techniques such as inspections, algorithm analysis, decision-table analysis, etc.
An inspection is an example of alternative QA processing. It's a critical examination of software development by a formal, group review of documents created during the development process. While most often it is program code that is inspected, program specifications, data diagrams, test plans and test data, and other documents can also be inspected. This technique, and the others listed, find errors more quickly than testing does, as they can be applied earlier in the development cycle. Even so, testing is the most common QA activity.
There are many different types of tests that are performed. Programmers do the unit testing on the programs they have written; systems testers verify systems and sub systems with system and integration tests, and users finally conduct the user, acceptance, or beta tests. Black box (or func­tional) testing makes sure the programs do what they're supposed to and white box (or structural) testing ensures the programs are technically correct. Regression testing is used to make sure new programming doesn't erroneously change part of a system that should be left alone.
Several methodologies are used, the most popular being Six Sigma, TQM (Total Quality Management), and CMM (Capability Maturity Model). TQM appeared first, and describes a meth­odology of quality control for software development. It works with ongoing refinements of the software product and continual feedback thus improving software in small increments. It was developed by IBM in the mid-1970s, and is part of ISO 9000 compliance. CMM was introduced by SEI (Software Engineering Institute) in the mid-1980s and defines five levels of effectiveness; initial (ad hoc and chaotic), repeatable, defined, managed, optimizing (piloting new development ideas and technologies.) Six Sigma is a term used by statisticians and engi­neers to describe a state of nearly zero defects. It's used in IT to define a methodology for quality control during software development and focuses on critical core processes to understand customer requirements and align work processes. It uses analytical tools to improve these processes.
