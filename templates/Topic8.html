Development Tools


It seems that as soon as people started to develop application programs, software appeared to help the programmer. Some development tools worked with the entire development life cycle, but other tools were developed to work with only one phase. This automation of the development process has been going on for many years and there now are many tools avail­able to developers. Development is often discussed as "high-level" which is done indepen­dently of the platform the system will run on, and "low-level" which requires knowing the specific run-time platform. The phases of the life cycle follow this designation, with analysis and design being high-level development and programming and testing being low-level. Knowledge of one or more of these tools is often a required skill of developers.
Low Level Development Tools
Low level tools are used to assist developers during the programming and testing phases of the life cycle. Work done during these phases requires knowledge of the platform or platforms the system will run on, the databases being used, and any necessary communications or networks that will be part of the system.
Application Development Environments (ADEs), Software Development Kits (SDKs), and Studios, or Integrated Program­ming Environment (IDE), is a software package that enables programmers to write, test, and document computer programs from a terminal. These tools are often called "devel­opment kits," and provide a visual environment for developers to do their work. These systems provide access to existing databases and contain report and program generators. This allows programmers to use point-and-click and drag-and-drop functions rather than writing lines of code. ADEs are also called RAD (Rapid Application Development) tools and GUI (Graphic User Interface) builders and are used to develop client/server and Internet applications. Many provide cross-platform capabilities (building applications that will run on different platforms). Some application development environments, especially those referred to as software development kits or studios, actually contain modeling tools to provide tools for most of the development cycle. Other development environments con­centrate on programming and testing, but interface with modeling tools.
Application development environments are heavily used for client-side development in client/server and Internet applications and in object-oriented development. They are also used for RAD development, report generation and query programming. These tools are easier to learn than are programming languages and are often called visual languages. Some of them really are "user friendly" and do not require any technical skills or back­ground. This enables businessmen/women to create many of their own reports and run their own queries. More and more programming is done with these tools.
Programming Utilities. This category is a catch-all for a variety of different programs that provide specific assistance to programmers. Most programming utilities are written by outside vendors, not by the computer manufacturer. These outside vendors have seen a need for support and often deal with computer systems from many vendors.
Many programming utilities are conversion tools. For instance, these programs can convert applications written to run under one operating system so they can run under another. Some utilities will convert JCL (Job Control Language) from one operating system to the JCL of another. And still others will convert online commands from one online system to an­other. Other programming utilities provide the needed interfaces so application programs can access software products (such as databases) from different vendors. These utilities can be important in a company that has more than one database or more than one communi­cation system. A programming utility will, for example, allow the programmer to use the same program to access both a DB2 database and an IDMS/R (интегрированная система управления данными) database.
Still others provide such functions as analyzing programs to make sure they follow company standards, analyzing data descriptions to eliminate redundancy, and defining critical datasets in a system to determine the resources (such as disk space, tape reels) necessary to process the data. Programming utilities provide just about any type of support for the programming function.
Report Generators A report generator, or report writer, is a program that generates formatted reports from simple input statements, thus eliminating the need to write com­plete programs to accomplish fundamental print tasks. Report generators exist for all types of computers because they save so much time. Rather than having to write a com­plete computer program, a programmer can simply provide a description of the data to be used and the way he or she wants the report to look; the generator will do the rest. These programs will generate the instructions necessary to do any desired totaling, aver­aging or percent calculation. In addition, they will take care of print spacing, header lines and/or trailer lines on reports, and page numbering. All JCLs and ADEs contain report generation functions.
Program Generators A program generator is a computer program that can generate all, or part, of the code necessary to build a program in a source language. Programmers provide input that describes the data the desired program will work with, how that data should be processed, and what the output should look like. The generator will then pro­duce the source statements that follow the syntax rules and use the vocabulary of the source language. Many of these generators do not produce the entire program, but can only handle part of it. Generators can usually produce source statements for data defini­tions, but often the statements that manipulate the data must be written by the program­mer. The most common program generators produce C, C++, or COBOL statements. Program generators are also called application generators and code generators, and are often part of ADEs.
Screen Editors Screen editors are computer programs that handle the input and modi­fication of programming statements from a terminal. These programs are similar to word processors and have features such as global changes (i.e. changing the character '"x" to "x\" everyplace it appears in a program), block movement of statements, and repetition of statements. In addition to handling programming source statements, screen editors also work with JCL statements. Screen editors also provide access to programming and data libraries so programmers can, through the screen editor, copy programs and/or data files, delete them, rename them, and so on.
Screen Painters/GUI Builders A screen painter is a program that will generate the code necessary to define a terminal screen for an application system from a sample of the screen. This means that the programmer can design the screen right on the terminal by moving the cursor around. Once he or she is satisfied with the way the screen looks, the program will generate the code to be incorporated in the online program. When a screen painter is also a GUI builder this means the developer can incorporate the GUI interface through the tool. Screen painters/GUI builders are part of application development envi­ronments or can be stand-alone tools used for online system development.
High Level Development Tools
High level tools are used during the analysis and design phases of development and do not require knowledge of hardware, databases, or specific communication or networking tools. Actually this is when decisions are made about which of these technologies will be used. Work done during analysis is often done by non-technical people who know a lot about how the business system actually works, but do not know computer technology.
CASE (Computer Aided Software Engineering). The tools developed to assist in the automation of the analysis, design, and project management functions are usually re­ferred to as CASE tools and have existed since the 1980s. Several things contributed to the emergence of CASE products. There was (and still is) an overwhelming demand for soft­ware. Most IT Departments have a three to ten year backlog of program requests and still spend most of their time maintaining existing systems. Also, while the cost of computers and associated hardware seems high, it doesn't approach the cost of software develop­ment. All these are reasons for the industry's look at ways to produce higher-quality pro­grams more quickly and with less effort. The overall goal of the CASE industry is to provide software that, from a specification of what a program is to do, will figure out how it should function and then write and test the program. In other words, CASE products were originally designed to automate the entire systems development cycle. While many of these tools still do this, the automation of the programming and testing phases is com­monly done by separate Application Development Environments and tools. Today's CASE products automate analysis, design, managerial functions such as planning and schedul­ing and provide automatic documentation throughout the system's development. Mod­eling tools are part of today's CASE environment.
Modeling and Modeling Tools. Modeling is used to specify, visualize, and document software systems. While it is required for large and complex systems (all data warehouses are modeled), it is valuable for any system development. Two definitive types of modeling exist: data and process modeling. Process modeling has two sub-sets - software and busi­ness process modeling. There are different methodologies and tools used for all three.
ER (Entity Relationship) modeling is the oldest and most common data modeling technol­ogy. It was introduced by Peter Chen in 1976 and identifies information as entities and ties these entities to each other through various rules that define relationships. As new technologies appeared, ER techniques and notational systems were incorporated and ex­panded. ER modeling is included in IE (Information Engineering), IDEF1X (Integrated Definition) (метод описания интеграции), and UML (Unified Modeling Language). Each of these systems has expanded the technology and each has its own notational system, but all are very similar and it's very easy to move from one to another.
Process modeling gets more involved - it's always harder to work with actions than it is with information, and that's what processes are - actions. Software processes can be programs, modules, objects, components. Modeling the software processes means defin­ing the executable segments - and when and how they will execute. Business processes, of course, are the actions of our businesses. For example, order entry is a business process. It consists of working with data (collecting it, updating it, etc.), but the main focus is on the action itself (answer the phone, ask specific questions, enter this data, etc.), when the action occurs (throughout the day, every Friday, at the close of business, etc.).
Several notational techniques are used in processing modeling, especially when creating software systems. Flowcharting simply uses a step-by-step graphical representation of a process that shows all decision making and the different steps taken at each decision point. Use Case shows the interaction of people, systems, and organizations with the processes. Action diagrams support parallel activities and their synchronization and can easily identify the process start for different user interests. Interaction diagrams describe the behavior of several objects in a single use case, and activity diagrams are good at showing the general sequence of actions for several objects and use cases. Simulation measures overall changes in results caused by the specific changes made to processes. These notational techniques are used in many modeling technologies including UML and IDEF (метод описания интеграции). UML is the dominant technology in the business world and innumerable tools sup­porting UML are in general use. IDEF is an older technology (it has actually been around for over twenty-five years) and is used much more by government agencies.
Business process modeling is the newest form of modeling and it is the one that could have the largest impact on IT. Business process modeling can be done with any process modeling tool and technology, but in fact new tools are being developed. Many of these tools are designed to be used by business men and women - not by IT professionals. This could cause significant changes in the way applications are developed. Two languages have already been developed - BPEL (Business Process Execution Language) and BPML (Business Process Modeling Language). Even more important, BPMN (Business Process Modeling Notation) has been introduced and is growing in use by both the business community and government agencies. This technology was developed specifically for business process modeling and is supported by over 20 IT companies including IBM and Microsoft. BPMN models are much easier for non-technical people to work with than those created with UML and IDEF.
Development Platforms - J2EE and .NET
A development platform is the technology used to develop and deploy (run) application programs. It actually includes three kinds of tools: development languages and utilities, middleware tools to connect application components, and execution programs to deploy the application. The two dominant development platforms are J2EE and .NET. Both plat­forms use object-oriented technology. Both platforms are used to build Web services. And the use of both platforms is growing by leaps and bounds. The platforms should be looked at through each of the three areas.
Development tools. This is the easiest - J2EE uses Java and .NET uses VisualStudio.NET, which includes C++ .NET, C# .NET and VB .NET (and support for other) languages.
Middleware (промежуточное программное обеспечение) tools. These provide the connectivity between application components including database access and multiple processing components. For example, an applica­tion that supports Internet purchases would have many components. When a Web visitor "proceeds to checkout," the application must access a database to make sure the selected products are available, access another database to store information about the purchaser. There undoubtedly are multiple processing components including a component that cal­culates shipping, another component that sends email confirmations to the purchasers, another component to total the purchase, etc. All these components must communicate with each other and both J2EE and .NET specify the middleware that defines the commu­nication. J2EE includes EJB (Enterprise JavaBeans) and JMS (Java Message Service) for com­ponent communication, and JDBC (Java DataBase Connectivity) for database connectivity. .NET uses MSMQ (Microsoft Message Queue), MTS (Microsoft Transaction Server), and C0M+ for component communication, and ODBC (Open DataBase Connectivity) for da­tabase access.
Execution, or Deployment. Both J2EE and .NET use an interpreter which works with generated code (bytecode in J2EE and C1L in .NET) and associated libraries and files to create an executable module. Java's components are interpreted by JRE (Java Runtime Envi­ronment) which is part of J2EE, and are then deployed by any Java compatible application server including WebSphere, WebLogic, JBoss, Enhydra, etc. Of course the Sun Java System Application Server is one of these servers. .Net applications are interpreted by CLR (Com­mon Language Runtime общеязыковая среда выполнения) which is part of the .NET Framework. CLR, however, also deploys the application which means that applications can only run on systems with the .NET Frame­work installed. This means .NET applications can only run on Windows platforms.
This is actually the major difference between the two development platforms. J2EE can only be used for Java development but can run on many execution platforms (the combination of the operating system and the computer). .NET can be used with many languages, but can only run on Windows platforms. Some other differences include: .NET is easier to learn and use; J2EE is an open specification while .NET is proprietary; and security is handled differently by the two systems. There are, of course, other differences, but one of the biggest growing areas of tool development is to create software that lets applications developed by either platform to interface with each other. While companies that use Unix, Linux, and mainframe systems are likely to use more J2EE, in all companies PC-based systems are likely to be devel­oped with .NET. It's no longer expected that companies have to pick one platform or the other. It works to use both.
