Development Life Cycle

Software engineering is a term that represents applying standards, rules, and methods to software development. Since the 1960s when computers first entered the business world, the need for some control over the way software was developed became apparent. Different methodologies and techniques are used in different environments and for developing different types of software. All software is developed following a life cycle of activity, in fact, producing anything follows this cycle – an architect knows this same cycle, as does an engineer creating a new machine. Systems (or buildings) start out with a definition of what is to be done, proceed to analysis (figure out what to do), then — design (determine how to solve the problem), actually doing it (programming), testing (make sure it works), and implementation (start using it for real).
Programming is, in its simplest definition, writing a sequence of instructions to be executed by a computer to solve a problem. Obviously, it is not quite this simple. Programs do not exist alone, but are part of a system. Developing a system is a cyclical activity that has several phases. In order to understand the job of programming, it is necessary to look at the entire project life cycle. Developing a computerized system, or even a single program, follows a cycle of activity that is referred to as the project life cycle, the programming life cycle, or the system development life cycle. Just about any combination of words can be used with the two important ones: life cycle. There is no definitive breakdown of exact steps, or phases, that make up the project life cycle, but all life cycles contain the same basic phases and activities. The activities in each phase will be discussed separately, but it is important to remember that systems development is a cycle. This means that the work done during each phase overlaps. With large systems, design will start before the detailed analysis is complete, programming starts during the design phase, and so on. Another aspect of the cyclical activity is that completing one phase can result in cycling back to a prior activity. A problem encountered in the programming phase often means going back to the original analysis to determine the cause.
