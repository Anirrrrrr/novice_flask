Languages


Computers are machines, and as intelligent as they might appear, they are devices that only recognize the presence or absence of electronic pulses. Picture a computer as being made up of thousands, even millions, of light switches. Each switch is either on or off — it either has an electronic pulse or it doesn't. These "switches" are called bits (binary digits) and are assigned numeric values. If a switch is on, it has the value 1; if it is off, it has the value 0 (or vice versa).
Each computer is designed with a machine language. Machine language, or machine code, is the combination of bits that the computer recognizes as instructions. For example, the bit combina­tion 1111 1010 could mean add, while 1101 0101 could mean compare. Each computer model has its own machine code. The code for an IBM mainframe is not the same as the code for a Unisys mainframe. In fact, it is not even the same as the code for an IBM microcomputer.
Originally, computer programs were written in machine code. The process was not only diffi­cult and time consuming; it was also inaccurate. Simply transposing ones and zeros would change an instruction completely. It is not surprising that computer languages such as CO­BOL were quickly developed.
Computer, or programming, languages were the first tools developed to automate at least some of the job of the programmer. A computer language is an English-like language that has a set vocabulary and syntax rules to cover such things as punctuation and word order.
Each language has a language translator, which is a program that takes the English-like instructions and converts them to machine code, thus creating a program the com­puter can execute. This means that there really are two versions of every program: the English language ver­sion used by the developer (called source code), and the executable machine language version, which is ex­ecutable. There are several different levels of computer languages ranging from low to high level. Low-level lan­guages are more difficult to learn and use, but create the most efficient programs. They take more time from the developer, but execute faster. The high level lan­guages are easier to learn and use, but the machine code programs generated are less efficient. Low-level languages are the closest to the actual machine language and are the most technical. The high-level languages are computer independent, work with any computer system, and require less technical knowledge by the programmer. Technical developers are more likely to work with low-level languages and applications developers with high-level languages and application development environments. Programmers today usually know many languages, and unless assembler language knowledge is required it is important to realize that a specific language is the least important of a programmer's skills. Languages can be learned quickly by experienced programmers as most of the activities involved in development, analysis, design, and testing, are independent of the programming language. Today programming languages are only one of the tools available to programmers. Tools exist to automate all phases of the development life cycle, and many tools are available espe­cially for the design and testing phases.
Many programming languages are in use today, and it's not unusual for a program­mer to have experience with over a dozen languages. Languages can be classified as low-level (machine specific) and high-level (machine independent.) Only two types of languages are low-level: machine language and assembler language.
All others are high-level and can be used in any machine environment as long as a translator pro­gram is available.
Assembler Languages
These languages are also called mnemonic languages, or mnemonic code. Assembler lan­guages are closely related to a specific computer and follow a one-for-one correspondence to its machine language. For example, a computer has as many as a dozen different instructions to add two numbers together. The instructions vary because of the location or size of the numbers. An assembler language must have a corresponding instruction for every one of these adds. Assembler languages have as many as 200 instructions that must be learned by the programmer. Because the instructions are so tied to the machine itself, assembler lan­guage is not transferable ['trænsf(ə)rəb(ə)l] between machines. If someone knows assembler language for an IBM mainframe, he or she cannot use this language to write a program for a Control Data machine. While most of the assembler programming is done instruction-by-instruction, pro­grammers can build macros by combining two or more assembler instructions. Working with macros is still low-level programming.
Assembler languages are rarely used in the application environment except for building em­bedded systems, but are used by the systems staff. Because the languages are so close to the actual operation of the machine, more efficient programs can be written although there is a tradeoff in programming time. Actually, extreme efficiency is usually not important in an application environment. When a company runs its payroll, the programs might take hours to run, but they are only run once every two weeks. It makes more sense to save time writing the programs than to save time in execution. The reverse is true with systems programs. A pro­gram written by a systems programmer might produce the date on reports. Because this program will be executed hundreds of times every day, speed of execution is critical.
Compiler Languages
Compilers are called third-generation languages and are used for the most critical program­ming. A compiler language is a general-purpose language and is not tied to any machine. Most of these languages were developed outside the computer industry. COBOL, the most popular of all the languages, was developed by the U.S. Navy. PASCAL was developed at a Swiss university. With these languages, syntax, rules, and vocabulary were defined, and then computer manufacturers wrote translator programs to take the pre-defined language input and create machine-specific output. This means that knowledge of a compiler [kəm'paɪlə] language is transferable. Although minor changes are often needed, a COBOL program written for an IBM machine can indeed be executed on a Control Data machine. All the programmer has to do is use the appropriate translator.
Compiler languages are much easier to learn than assemblers [ə'sembləz] , and programs take less time to write. A compiler, for example, will have a single add instruction. The translator chooses which of the machine instructions to use, and this saves time for the programmer both in learning the language and in writing programs. Compiler languages are heavily used in appli­cations programming and some, such as C and PASCAL, are efficient enough to be used in systems programming. Object-oriented, interpreters, and scripting languages are all varia­tions of compiler languages.
Object-Oriented Languages Specific compiler languages were developed to work with object technology and work with object classes and libraries. C++ and Java are the most common of these languages although there are many, all of which are used by some companies.
Interpreters. An interpreter is any programming language that works by translating the English language statements into machine code and immediately executing the code. That is, the translation and execution are both done every time the program needs to run. Interpreters are very flexible and allow the devel­oper to immediately see results. Scripting languages are usually interpreted languages, as are some compilers such as Java and Basic.
Scripting Languages There is debate over whether scripting languages are a subset of compiler languages or a separate category, but whether they are or not, they have the following characteristics. These languages are specialized programming languages that pro­cess text strings, search files, databases ['deɪtəbeɪs] and indexes, and generate reports. They do not have all the features of other languages and would not be used for critical programming, but are excellent for input/output intensive situations. They are easier to use than any other languages and have gained popularity with the growth of the Internet and Web development. Scripting languages have been used with operating systems for many years, particularly with Unix. Unix shell languages are scripting languages, as is the Unix extension Perl.
Query languages
Many different languages fall under the category "query languages." A query is a program that produces a report. A report is anything that goes to paper or a terminal screen. This means the information is non-permanent - permanent data must be stored on tape or disk in the computer world. Only data that is accessible by another program is permanent. Report programs actually are simply defined. These programs access a data collection (usually a database) and select specific data or summarize data. Selection can be listing records "pro­duce a list of employees in the accounting department" or information from a record "show me name and address for the employee whose social security number is 123456789." Sum­marization is one of three things: totaling information, calculating percentages, or calculating averages. The report (the selection or the summary) can go to paper or to a terminal screen. Many people classify scripting languages as query languages, and this category also includes 4GLs and visual languages.
4GLs (Fourth Generation Languages)
4GLs, or query lan­guages, are also called user-friendly, query, and nonprocedural languages. These languages are almost free-form in structure and have few rules and predefined vocabulary. Because they were designed to be used by nonprogrammers (hence the title user-friendly), the translator program works from a simple definition of what the output should be and does not require step-by-step instructions to achieve the result.
4GLs concentrate on instructions to query databases. A query to a database can be asking for a response back to a terminal screen (such as "Show me John Johnson's checking balance") or a com­plete report ("Produce John Johnson's monthly checking account balance").
These lan­guages are easy to use and programs can be written very quickly using them. They are excellent for queries and reports, but do not have the control of assembler, or even com­piler languages. They are used mostly by applications programmers for report generation and to develop Internet and client/server applications. While learning and using a 4GL is simpler, than working with a compiler language, it still requires technical knowledge and these languages did not prove to be "user-friendly." SQL (Structured Query Language) is the most dominant query language.
Visual Languages Visual languages are also called ADEs (Application Development Envi­ronments). There are systems that allow developers to use a GUI (Graphic User Interface) to create programs rather than writing code statements. These "languages" also are used for testing, so really should be classified as development tools, rather than as languages. VB (Visual Basic) and Delphi are both visual languages.
Some Specific Languages
There are three languages which are used extensively and worth looking at specifically: 1) SQL (Structured Query Language), which is the standard query language used to access rela­tional databases; 2) UML (Unified Modeling Language), which is used to build design models; and 3) XML, (extensible Markup Language) which is a standard interface language.
SQL (Structured Query Language)
SQL is a 4GL that was originally developed to query relational databases and has become an industry standard. Most databases from all vendors are accessible through SQL, and it is used in mainframe, midrange, and desktop environments. Its capabilities have been extended to access databases of all types, and even data stored in file structures. More and more, SQL knowledge is becoming part of the necessary skill base for all programmers in business environments. Because SQL has been accepted throughout the entire computer industry, it is dominantly used in any multi-vendor or multi-platform environment. Client/server and Internet applications require that different systems be able to function together, and SQL is used in the server-side processing.
SQL is recognized by ANSI (American National Standards Institute) and the current version in use is SQL2. This doesn't necessarily mean people use the numeric designation, but it's worth noting because of the existence of SQL3. SQL3 is the language used to access objects stored in an object-relational database.
UML (Unified Modeling Language)
UML is both a development methodology and notational format used in object analysis and design and works with all object-oriented methodologies. Using UML (Unified Modeling Lan­guage), developers define a three-tiered model of the application: user interface, business logic and database in which they describe class (of objects), object, association, responsibility, activity, interface, use case, package, sequence, collaboration, and state. Booch, Rumbaugh, and Jacobson, three of the major developers of object-oriented technologies, created this single modeling system in the mid 1990s, and it is now an accepted standard of the Object Management Group (OMG).
XML (Extensible Markup Language)
XML is a markup language that enables the definition, transmission, validation, and interpretation of data between applications and between organizations. It is a product of W3C (World Wide Web Consortium) and is vendor, platform, application, language and product independent. This defini­tion really does define a technology, not just a language, and XML is used in many different ways.
A markup language is one that uses tags to specify content and/or appearance of printed or displayed information. HTML is the most common markup language, and it is used to specify the look of information on Web pages. In HTML:
The code: would show:
<l>be sure to check</l> (italic) be sure to check
<B>NEW RELEASES!</B> (bold) NEW RELEASES!
While HTML provides all the allowable tags, XML goes beyond this (hence the name eXtensible) in that it permits the definition of additional tags through a DTD (Document Type Definition). These new tags can than be used in documents, text files, Web pages, and databases. For example, if a tag <CustomerName> existed, that tag could be used in the Web page when the visitor filled in a name field, used in a database definition so the information would automatically be transferred to the database, then used in a report definition that would provide a list of new customers to appropriate management.
A set of defined tags is called a vocabulary, and vocabularies are being defined by industries, associations, scientific fields, etc. The benefits of industry-defined vocabularies are many. If the insurance vocabulary includes the tag <policynumber>, and Prudential, Allstate, and State Farm all use that tag, a Web site could find policies from all three companies. CML (Chemical Markup Language) was one of the first XML vocabularies, and covers many chemical disciplines. HRML (Human Resources Markup Language) defines HR standards and allows corporate applications, job board Web sites, ATSs (Applicant Tracking Systems), and consulting firms and agencies to share job listings. In addition to these vertical vocabularies, horizontal vocabularies that can be used across industries have been accepted by W3C. One of these, VoxML (Voice Markup Language) allows developers to add speech interfaces to their Web applications and can be used by developers in any industry. Literally hundreds of vocabularies have been defined and published.
XML is an important tool and is fast becoming a standard skill for almost all IT developers. It's obviously used in Web development, but has additional uses. The legal field is starting to use XML to search the volumes of case work and legal issues that build a case. The Gene Ontology Consortium is producing a dynamically-controlled vocabulary that can be applied to all or­ganisms even as knowledge of gene and protein roles in cells is accumulating and changing. XML will allow automatic incorporation of the new knowledge. And, the dominant use in today's corporate world is to build an infrastructure between internal applications! Compa­nies have many applications - home-grown, ERP (Enterprise Resource Planning) systems, ven­dor supplied, eApplications, and Web front-ends. And, too often, these systems can't exchange information and processing without lengthy programming. Corporations are using XML to solve this problem. Estimates state that 70-80% of all development will be using XML. XML is poised to become a necessary skill for almost all developers, and one of the most dominant tools in all of IT.
